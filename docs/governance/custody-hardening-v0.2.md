\# Custody Hardening v0.2



\## Purpose



v0.2 introduces custody hardening for Governed Consequence Routing.



v0.1 demonstrated that governance events can be promoted into decision envelopes, schema-validated, hash-linked, independently verified, and mutation-tested.



v0.2 strengthens the proof surface by adding clearer custody artifacts around verification.



\## Core Distinction



v0.1 proves local tamper evidence.



v0.2 begins custody-aware verification.



This does not mean legal admissibility, external notarization, or production-grade custody.



It means the project now records enough verification context to make the proof artifact easier to inspect, reproduce, and challenge.



\## Custody Goal



A verifier should be able to answer:



\* What envelope chain was verified?

\* Which schema was used?

\* Which verifier version was used?

\* What record hashes were checked?

\* What chain head was produced?

\* What mutations were tested?

\* Which mutation checks were detected?

\* What tamper-evidence mode was used?

\* Was the proof unkeyed or HMAC-backed?

\* If HMAC-backed, which `key\_id` was declared?

\* What proof boundary applies?



\## v0.2 Artifacts



v0.2 should introduce:



\* verification bundle export

\* chain summary report

\* mutation report

\* verifier metadata

\* schema hash capture

\* chain head capture

\* optional HMAC custody mode

\* explicit proof-boundary language



\## Verification Bundle



A verification bundle is a portable proof artifact generated by the verifier.



It should include:



\* bundle version

\* creation timestamp

\* verifier version

\* schema path

\* schema hash

\* envelope source path

\* envelope count

\* first record hash

\* final record hash

\* tamper-evidence mode

\* key identifier, if applicable

\* verification result

\* mutation-check result

\* detected mutation classes

\* proof boundary statement



The bundle is not a legal certificate.



It is a reproducible verification artifact.



\## HMAC Custody Mode



v0.2 may support HMAC-backed custody checks.



HMAC mode must obey these rules:



\* no HMAC secret is committed to the repository

\* secrets are provided only through environment variables or local ignored files

\* the envelope records only `key\_id`, never the secret

\* verifier output may confirm HMAC verification status

\* test fixtures may use clearly marked demo keys

\* demo keys must never be represented as production custody keys



\## Proof Boundary



v0.2 does not claim:



\* legal admissibility

\* third-party validation

\* external notarization

\* production custody

\* secure key management

\* non-repudiation

\* enterprise audit readiness



v0.2 claims only:



\* clearer custody metadata

\* reproducible verification bundles

\* stronger local tamper-evidence story

\* optional HMAC-backed verification path for local demonstration



\## Governance Principle



The custody layer strengthens the record.



It does not change the authority model.



The core rule remains:



> The model may propose. It may not authorize.



Policy decides.



The decision envelope records.



The verifier checks.



The custody bundle explains what was checked.
